# 自动科研与闭环系统

这一页聚焦“从研究想法到实验反馈再到报告输出”的自动科研闭环。相比 Deep Research 系统，这一类项目更强调任务分解、实验执行、评测循环、自我修正与长期研究进展。

## 4. 常见闭环结构

1. `问题定义`
   输入通常是研究方向、任务目标、限制条件和已有先验。
2. `文献与证据收集`
   通过检索代理、论文问答系统、知识图谱或外部数据库形成证据底座。
3. `假设与方案生成`
   使用 ideation agent、review agent 或 hypothesis generator 形成候选方案。
4. `代码与实验执行`
   把候选方案转成代码、配置、实验脚本和运行任务。
5. `评测与反馈`
   将指标、错误日志、失败案例、消融结果反馈给计划器或评论代理。
6. `报告与沉淀`
   形成研究笔记、实验记录、草稿、预印本或共享知识资产。

## 5. 可直接抽象到科研工作流的设计原则

| 原则 | 说明 |
| --- | --- |
| 计划与执行分层 | planner 负责课题分解，executor 负责代码、检索、运行和写作，可减少“大一统代理”失控。 |
| artifact 显式化 | query pack、paper card、hypothesis bank、experiment spec、run log、draft 都应成为独立对象。 |
| evaluator 独立存在 | 评测器、审稿器、novelty 审计器与执行器分开，更容易做回溯与错误归因。 |
| 环境隔离 | 自动实验系统通常需要沙箱、容器或远程工作区，避免在主环境直接执行不可信代码。 |
| 人工 gate | 在课题确认、实验预算、结果解释和投稿前增加人工确认节点，通常更稳。 |

## 6. 与评测资源的连接

自动科研系统很容易被“demo 效果”误导，适合同时搭配下列评测资源：

- [Aviary](https://github.com/Future-House/aviary)：面向科学任务的 language agent gym。
- [LAB-Bench](https://github.com/Future-House/LAB-Bench)：面向生物科研基础能力的评测集。
- [BixBench](https://github.com/Future-House/BixBench)：面向 computational biology 的 agent benchmark。
- [MLE-bench](https://openai.com/index/mle-bench/)：机器学习工程与实验执行相关 benchmark。
