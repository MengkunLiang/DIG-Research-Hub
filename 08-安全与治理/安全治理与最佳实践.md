这份资源库越往后越重要的一件事，是避免科研智能体“看起来很会干活，实际上在高风险环节胡来”。

## 2. 对 ResearchOS 特别重要的治理原则

### 原则一：文献结论必须可追溯

最低要求：

- 保留 DOI / arXiv / OpenReview / OpenAlex 等原始链接
- 关键 claim 要能回到原文证据
- novelty 判断不能只靠二手总结

### 原则二：实验结果必须可复现

最低要求：

- 保留配置文件
- 保留运行日志
- 保留指标产物
- 保留环境和依赖信息

### 原则三：对外输出必须设人工 gate

尤其包括：

- 摘要与结论
- reviewer response
- 投稿包
- 给导师或合作者的正式版本

## 3. 具体到工具层的建议

| 工具类型 | 建议 |
| --- | --- |
| 学术检索 MCP | 优先官方或活跃实现，不直接信任单一结果。 |
| Zotero / GitHub / Notion 类 MCP | 默认最小权限，只暴露必要目录和项目。 |
| 网页搜索 MCP | 作为补充信息渠道，不替代学术数据库。 |
| 自动写作工具 | 输出必须挂接引用和证据，否则只能算草稿。 |
| 代码执行工具 | 默认隔离执行环境，避免污染主环境。 |

## 4. 推荐你在系统里固定的几个治理检查点

1. `topic freeze`
2. `novelty gate`
3. `experiment budget gate`
4. `result sanity check`
5. `submission gate`

这几个点一旦固化，系统就不容易从“科研助理”滑向“高风险自动化”。

## 5. 一条简单但实用的经验

如果一个能力会同时满足下面三条，就默认把它放进高风险区：

- 能访问外部世界
- 能修改重要资产
- 能直接形成对外输出

这类能力必须加权限约束、日志留痕和人工复核。
